# project overview
Important: This page does two things: (1) provides an overview of the project and (2) provides 10 'easy' points for simplying adding yourself to a project group in Canvas. You MUST do this, even if you are working alone, since Canvas uses the groups to allocate points for the project components. Failure to add yourself to a Canvas group before the deadeline complicates the grading workflow andwill result in a 0 out of 10 for the points associated with this page.

Three sentence summary of the project: Form a group of 1 to 4 members, design a project using Deep learning that does something fun, interesting, creative, ambitious, and challenging. Host your code on GitHub, create a 4 to 6 page summary (longer is fine) document what you did (including images and bibliography), then present your goals, methods, and results in a 15 minute presentation at the end of the semester.

Overview
The projects MUST use deep learning and neural networks as a core component of the workflow.
The project MUST be a challenging problem, beyond what is done in the homework. It is okay to use the homework as a starting point but the project should take it to a significantly higher level.
The implementation must be in python.
The final lecture and lab time-slot will be dedicated to allowing the groups to present their solution of the group project.
Kaggle data and projects are allowed. In fact, you are more than welcome to try to compete in a currently active competition. However, the project must be a difficult one. For example, heavily studied educational competitions such as MNIST are not allowed.
Deliverable components
Each group will be responsible for the following components
Code: Provide an original solution to the problem (hosted on Github).
Write-up: Provide a concise 4 to 6 page report document about what you did (longer is fine).
Slides and presentation: Present your goals, methods, and results in a 15 minute presentation at the end of the semester
Group selection
You are free to choose your own group members. If the course has multiple sections then you can choose members across sections. However, if you do you will need to be flexible about which section you present in, during the final week of class.
Group "self sign-up", in Canvas, under the "people" tab, will be opened and announced in the first two weeks of the semester.
If you have no preferences and are not in a group after the 3rd week of class, then you will need to work alone
Minimum group size is 3 and maximum is 5.
This is a large class, so some groups may need to present their results remotely during the final week. This will depend on scheduling and group sizes.
Submission and due dates
Upload the following to Canvas before the start of the section in which your group is presenting during the last week of class
PDF of your report
URL to the GitHub repo hosting the project code
PDF of slides
Grading
The project will be evaluated based on
Quality of presentation and slides
Holistic score on the quality of the overall project design and implementation
For example, was the project sufficiently ambitious, was it attempted in a serious and professional fashion, were the results and methods acceptable, ... etc.
Holistic grading of the quality of submitted code and generated results stored on Github
Your average peer evaluation from group members
You will be graded on the quality of the attempt, not the quality of your model, for example if you try something very ambitious but your accuracy is low then you won’t be penalized for the low accuracy

# project code
Assignment Description:

In this portion of the assignment, you will develop a set of scripts and/or modules to support your project. Your task is to create, refine, and document your code and project documentation in a GitHub repository provided through GitHub Classroom. You will work collaboratively, treating this project as you would a software development endeavor in a professional environment.

When possible, please use the .ipynb format for your code, it makes it easier to review and grade.

General Expectations

GitHub Classroom: You must host your project on the assigned GitHub Classroom repository. A project invitation link will be distributed early in the semester.
Collaboration: Work together, but designate roles as needed (e.g., an “admin” to act as a lead). Use branches, pull requests, team reviews, and merges to manage your development process.
Frequent Commits & Documentation: Make regular (daily or weekly) commits. Part of your grade will come from how effectively you use GitHub’s collaborative tools (issues, pull requests, merges, comments, etc.).
Organization & Documentation: Keep a consistent folder structure, meaningful file names, and sufficient documentation to allow others to understand and replicate your work.
Final Submission: At the end of the semester, submit both:
A compressed archive of your final GitHub repository (without data or other large files, folder should be less than 50MB)
A link to the public GitHub repository page.
What Makes Good Code?

Readability
Use clear, consistent naming conventions for variables, functions, and classes.
Maintain an easy-to-follow structure with logical blocks of code and comments where needed.
Modularity & Reusability
Break down complex tasks into smaller functions or modules.
Write code that can be reused or extended for similar tasks in the future.
Efficiency
Strive for clean, optimized solutions.
Avoid redundant calculations or unnecessary data processing.
Robustness
Handle potential errors or edge cases gracefully.
Use version control effectively to track changes and revert if needed.
Testing & Validation
Include tests (where feasible) to verify correctness.
Document how others can run and interpret your tests.
What Makes a Good GitHub Repository?

Clear Repository Structure
Organize files into meaningful directories (e.g., src/, data/, docs/).
Place large datasets or generated files in .gitignore if they are not needed for versioning.
Comprehensive Documentation
Maintain a thorough README.md describing the project’s purpose, setup instructions, and usage examples.
Provide instructions for running and testing the code.
Branching & Pull Requests
Create separate branches for new features, bug fixes, or experiments.
Use pull requests to merge changes into the main branch after team reviews.
Frequent & Informative Commits
Commit changes often with clear messages (e.g., “finalized conclusion section of poster and re-rendered”).
Reference issues or tasks in commit messages to enhance traceability.
Team Collaboration Tools
Assign issues to team members, tag pull requests for reviews, and comment on code changes.
Keep an open line of communication via GitHub’s discussion or issue-tracking system.
Remember: This project is graded not only on the technical merits of your code but also on your use of GitHub’s collaborative features. Aim for professional-level organization and team coordination throughout the semester.
